## 1. Где находится

На любой странице с шапкой приложения (`AppLayout`), **справа**, между селектором проекта и кнопкой **«Тема»**:

- иконка **Activity** (пульс);
- подсказка: **«Журнал задач»**;
- бейдж с числом — сколько задач в статусе `pending` / `running` по текущему проекту.

Панель открывается поверх интерфейса (`z-index: 1300`, выше карты и toast). Закрытие: крестик, повторный клик по иконке или клик вне панели.

Журнал привязан к **выбранному проекту** в шапке. Без проекта кнопка не показывается.

## 2. Не путать с «Администрирование → Журнал задач»

| | **Шапка: Журнал задач** | **Админ: `/admin/jobs`** |
|--|-------------------------|---------------------------|
| Кто видит | Все с доступом к проекту | Только администратор |
| Область | Текущий проект | Все проекты в системе |
| Назначение | Отладка расчётов, экспорт JSON запросов/ответов | Операционный мониторинг очереди, отмена зависших job |
| Данные | HTTP-шаги + последние jobs проекта | Глобальный список из БД |

## 3. Что показывается

### 3.1 Операции с HTTP-шагами (`http_flow`)

Одна карточка на пользовательское действие, внутри — цепочка запросов с телами **Запрос** / **Ответ**.

Примеры:

| Название в журнале | Действие на карте |
|--------------------|-------------------|
| Построение сети автодорог | Режим **«Сеть»** → «Построить» |
| Соединение автодорогами | Группа → **«Соединить автодорогами»** |

### 3.2 Фоновые задачи (`project_job`)

Записи из таблицы `project_jobs`: тип, статус, **Payload** (что ушло в очередь), **Result** (что вернул воркер).

Типы:

| `job_type` | Подпись в UI |
|------------|--------------|
| `autoroad_connect` | Автодороги / сеть |
| `import_file` | Импорт файла |
| `sand_logistics_analyze` | Логистика песка |
| `poi_analyze_all` | Анализ POI |

Список подгружается с API `GET /api/v1/projects/{id}/jobs?limit=30`; активная задача опрашивается каждые ~2 с.

---

## 4. Какие HTTP-запросы попадают в журнал

Логируются **POST/PUT/PATCH** по расчётным и импортным endpoint’ам:

- `/autoroad-network/request`, `/compute`, `/apply`, `/plan`
- `/infrastructure/autoroad-connect`
- `/sand-logistics/analyze`
- `/pois/analyze-all`, `/pois/{id}/analyze`
- `/import/...` (в т.ч. async)
- `POST /projects/{id}/jobs`

Ответ **202** с `job_id` дополнительно создаёт запись `project_job` и связывает её с предшествующими HTTP-шагами (если были).

---

## 5. Два POST при «Соединить автодорогами» — это нормально

Legacy-поток **не** использует JSON-конвейер «Сеть». При одном нажатии «Соединить» и подтверждении «Применить» уходят **два** запроса на один URL:

```text
POST .../infrastructure/autoroad-connect  { object_ids, dry_run: true }   ← preview
POST .../infrastructure/autoroad-connect  { object_ids, dry_run: false }  ← запись в БД
```

Backend **оба раза пересчитывает** план; отличие только в `dry_run`. В журнале это **одна** карточка «Соединение автодорогами» с **двумя** шагами.

Режим **«Сеть»** (рекомендуется для полной перестройки):

```text
request  → NetworkPlanRequest
compute  → NetworkPlanResponse
apply    → тот же plan (без повторного compute)
```

Подробнее: [autoroad-network-instruction.md](./autoroad-network-instruction.md) §4.

## 6. Экспорт JSON

- **Скачать JSON** (иконка на карточке) — один файл с записью: шаги HTTP и/или `payload` / `result`.
- **Экспорт всего** (внизу панели) — все записи текущего проекта из журнала сессии.

Имена файлов: `task-{тип}-{id}-{дата}.json` или `task-log-{project}-{дата}.json`.

Данные журнала хранятся в **sessionStorage** браузера (`taskLog:{projectId}`, до ~30 записей). После полного закрытия вкладки история может пропасть; список jobs в БД подтягивается при открытии панели.

## 7. Действия в панели

| Действие | Когда доступно |
|----------|----------------|
| Раскрыть карточку | Всегда |
| Скачать JSON | Всегда |
| **Отменить** | Job в статусе `pending` (не для «живого» `running`, кроме зависших по таймауту на backend) |
| Экспорт всего | Есть хотя бы одна запись |

Отмена: `POST /api/v1/projects/{project_id}/jobs/{job_id}/cancel`.

## 10. Частые вопросы

**Почему в журнале два одинаковых `object_ids`?**  
Скорее всего legacy **«Соединить автодорогами»**: первый шаг — preview (`dry_run: true`), второй — apply. Смотрите поле `dry_run` в JSON запроса.

**Почему нет записи после импорта CSV?**  
Синхронный импорт без job в очередь может не попасть в whitelist. Async-импорт (`.../import/.../async`) и ответ 202 — попадает.

**Почему apply «Сеть» не дублирует compute в JSON?**  
На apply уходит готовый `plan` из шага compute; планировщик на apply **не вызывается** повторно.

**Журнал пустой на новом проекте**  
Нормально, пока не запускали расчёт. Откройте панель после «Построить» / импорта / анализа.
