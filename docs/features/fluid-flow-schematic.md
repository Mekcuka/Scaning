# Схема потоков (PFD)

Вкладка **«Потоки»** (`/flows`) показывает схематическую диаграмму движения фаз от выбранной **точки интереса (POI)** по правилам нефтегазовой отрасли и по **графу инфраструктуры** проекта.

**Связанные документы:** [calculation-logic-flow.md](../calculations/calculation-logic-flow.md), [map-objects-and-spatial-calculations.md](map-objects-and-spatial-calculations.md), [input-parameters.md](../product/input-parameters.md).

**Дата актуализации:** май 2026.

---

## 1. Назначение

- Визуализировать **разделение продукции** на кусте: нефть, вода, газ (упрощённая трёхфазная модель MVP).
- Показать **куда направляется каждая фаза**: подготовка нефти, утилизация газа, закачка воды, маршрут по трубопроводам до приёмных объектов.
- Использовать **ту же сеть**, что хранится в БД после `build_network_from_lines` (пересборка при создании линий, импорте, расчёте логистики или `POST .../networks/build`; **на карте не рисуется**).
- Допускать **редактирование макета** (позиции, связи, пропускная способность) с сохранением в БД; расчётная топология пересчитывается при каждой загрузке.

---

## 2. Источники данных

| Источник | Роль |
|----------|------|
| POI (`points_of_interest`) | Имя куста, `fluid_type`, инженерные параметры `eng_*`, **`water_injection_volume`** (объём закачки воды, тыс. т/год) |
| `active_fluids` (`fluid_routing.py`) | Какие ветки PFD строятся (отдельно от `apply_engineering_rules` для анализа окружения) |
| `infrastructure_networks` + nodes/edges | Топология после `POST .../networks/build` |
| Подтип линии на ребре | Через `infrastructure_object_id` → `oil_pipeline` / `gas_pipeline` / `water_pipeline` |
| Терминальные точки | Узлы сети, привязанные к объектам карты (НПЗ, ГКС, **БКНС** и т.д.) |
| `infrastructure_objects.properties` | `throughput_capacity_annual`, `capacity_unit` — с карты и для терминалов на авто-схеме |
| `poi_flow_schematic_layouts` | Пользовательский макет (позиции, ручные блоки, сохранённые лимиты) |

**API:**

| Метод | Путь | Назначение |
|-------|------|------------|
| GET | `/api/v1/projects/{project_id}/pois/{poi_id}/flow-schematic` | Расчётная или объединённая с layout схема |
| PUT | `.../flow-schematic` | Сохранить layout |
| DELETE | `.../flow-schematic` | Удалить layout → снова чистый расчёт |

## 3. Правила ветвления (MVP)

### 3.1 Сепарация

От POI всегда идёт связь к узлу **«Сепарация»**, далее — активные ветки:

| Ветка | Условие |
|-------|---------|
| **Нефть** | `fluid_type = oil` |
| **Вода** | **всегда** для нефтяного POI (`fluid_type = oil`); маршрут зависит от `eng_injection` |
| **Газ** | `fluid_type` ∈ {`oil`, `gas`} |

> Для **газового** POI ветка «Вода» не строится.

### 3.2 Нефть

1. При `eng_oil_preparation ≠ mfns` — узел **«Подготовка»** (МКОС, БМУПН, …).
2. При `eng_transport = auto` — термин **«Автовывоз»** (без поиска по сети).
3. При `eng_transport ∈ {pipeline, marine}` — BFS по рёбрам `oil_pipeline` до `refinery` или `oil_pumping_station`.

### 3.3 Вода и объём закачки POI

Параметр **`water_injection_volume`** (тыс. т/год) задаётся на POI и **не скрывает** ветку «Вода». Он отображается во **всплывающем окне блока «В пласт»** (не на фазовой ветке «Вода»).

| `eng_injection` | Маршрут на PFD |
|-----------------|----------------|
| **`local`** | Вода → **«В пласт»** (без водопровода) |
| **`centralized`** | Вода → *(водопровод)* → **БКНС** (`ground_pumping_station`) → **«В пласт»** |

**Централизованная закачка — детали:**

- BFS по `water_pipeline` до **только БКНС** (`ground_pumping_station`), не до УПСВ/ДНС.
- После БКНС — обязательный узел утилизации **«В пласт»**.
- Если сеть не построена или маршрут до БКНС не найден — показывается **схематическая** цепочка **БКНС → В пласт**; при отсутствии реального пути — предупреждение `no_path_for_water`.

**Локальная закачка:** вода сразу на **«В пласт»**; объём закачки в popover блока «В пласт»: значение с POI или «не задан» при `water_injection_volume = 0`.

> **Анализ окружения (карта)** по-прежнему использует `apply_engineering_rules`: централизованная закачка активирует `water_pipeline` всегда; локальная — только при `water_injection_volume > 0`. PFD и анализ **разведены намеренно**.

### 3.4 Газ

- `eng_gas ∈ {well, flare}` — узел утилизации («В пласт» / «Факел»), без газопровода.
- Иначе — BFS по `gas_pipeline` до `gas_processing`, `gtes`, `gpes` или `vies`.

## 4. Построение маршрута по сети

1. На карте импортируют линии и точки, затем **«Сеть»** (`build_network_from_lines`).
2. Для фазы выбирается **ближайший к POI узел**, инцидентный ребру нужного типа трубы.
3. **BFS** по рёбрам с допустимым подтипом до терминального узла (кратчайший по числу переходов).
4. На схеме: агрегат **«Нефтепровод (N км)»** / **«Водопровод (N км)»** + терминал с именем объекта.

**Терминалы воды на PFD:** только **`ground_pumping_station` (БКНС)** — см. `FLUID_TERMINAL_SUBTYPES["water"]` в `app/services/fluid_routing.py`.

Код: `app/services/fluid_flow_schematic.py`, `app/services/flow_schematic_merge.py`, константы: `app/services/fluid_routing.py`.

## 6. Редактирование схемы

Схему можно **рисовать и править** на вкладке «Потоки»:

| Инструмент | Действие |
|------------|----------|
| **Выбор** | Перемещение блоков, выделение, Delete — удалить |
| **Связь** | Тянуть от выхода блока ко входу другого; тип флюида — в списке рядом |
| **Блок** | Выбрать тип, кликнуть на поле схемы |
| **Подпись…** / двойной клик | Переименовать блок |
| **Авто-раскладка** | Dagre слева направo |
| **Сохранить** | `PUT .../flow-schematic` → таблица `poi_flow_schematic_layouts` |
| **Пересчитать** / **Сброс** | `DELETE .../flow-schematic` → удалить layout и вернуть расчётную топологию; кнопка активна всегда («Пересчитать» для расчётной, «Сброс» — если был сохранён пользовательский макет) |

### Слияние layout и расчётной топологии

При загрузке сохранённой схемы (`source = custom` в layout):

1. Сначала строится **актуальная расчётная** схема по POI и сети.
2. Если **топология** (набор системных узлов) изменилась — подставляется расчётная цепочка (например, новая **БКНС → В пласт**).
3. Из layout сохраняются: **позиции** (`position_x`, `position_y`), **пропускная способность**, пользовательские блоки (`kind = custom`).

Код: `app/services/flow_schematic_merge.py`.

### Пропускная способность

У блоков оборудования при наведении — **всплывающее окно** (белый фон) с годовой пропускной способностью:

| Тип блока | Поведение |
|-----------|-----------|
| **«Нефть» / «Вода» / «Газ»** (`fluid_branch`) | Только **поток** по фазе; лимит не задаётся |
| **«В пласт»** (вода, `utilization`) | **Объём закачки** с POI (`water_injection_volume`), не поле «Пропускная способность» |
| Оборудование, трубопроводы, терминалы | **Пропускная способность** (тыс. т/год или тыс. м³/год); редактируется в popover, сохраняется в layout автоматически по «Применить» |

**Источники лимитов:**

- оценка из `flow_capacity.py` при авто-расчёте;
- `properties.throughput_capacity_annual` объекта карты — для терминалов, связанных с сетью;
- ручной ввод на PFD — в layout POI.

Поля API: `throughput_capacity_annual`, `capacity_unit` (`thousand_t_per_year` | `thousand_m3_per_year`).

### Поток по цепочке и перегрузка

1. **POI** — исходный годовой поток (плановый дебит или заданное значение).
2. По направленным связям поток передаётся на следующие блоки; после **сепарации** — по веткам с долями из `flow_capacity.py` (нефть 85%, вода = `water_injection_volume`, газ — по формулам).
3. Если `flow_annual` > `throughput_capacity_annual` (в тех же единицах), узел получает `over_capacity: true` и **подсвечивается красным**.

Поля API: `flow_annual`, `flow_unit`, `over_capacity` (пересчитываются при каждом GET, **не сохраняются** в layout).

## 7. Пропускная способность на карте

На **точечных** объектах инфраструктуры в `ObjectDetailPanel` доступно поле **«Пропускная способность»** (`properties.throughput_capacity_annual`, `properties.capacity_unit`).

**Исключения** (поле не показывается): `node`, `oil_pad`, `gas_pad`, `sand_quarry`, `substation`, `vies`, `gtes`, `gpes`.

Единицы: **тыс. т/год** (по умолчанию), **тыс. м³/год** — для `gas_processing`.

При авто-построении PFD терминал **БКНС** подтягивает лимит из `properties` связанного объекта карты.

Подробнее: [map-objects-and-spatial-calculations.md](map-objects-and-spatial-calculations.md) §1.6.

## 9. Ограничения MVP

- Одна схема на **один POI** (не сводная по проекту).
- Направление на ребре = порядок вершин в геометрии линии, не гидравлический расчёт.
- Не показываются: `autoroad`, `power_line`, `methanol_pipeline`.
- Редактирование PFD не меняет карту и граф инфраструктуры — только макет и сохранённые лимиты узлов.
