## 0. Навигация (реализовано)

После входа доступно боковое меню (`AppLayout`).

**URL проекта:** id в **конце** пути — `/map/{projectId}`, `/matrix/{projectId}`, … Пример: `http://localhost:5173/map/a1b2c3d4-…`. Без id: `/login`, `/register`, `/projects`, `/projects/:id`, `/admin/*`. Старые адреса (`/map`, `/{projectId}/map`, …) редиректятся на `/раздел/{projectId}`.

| Раздел | Маршрут | Кто видит |
|--------|---------|-----------|
| Дашборд | `/dashboard/{projectId}` | все роли |
| Проекты | `/projects` | admin, analyst, viewer |
| Карта | `/map/{projectId}` | все |
| Кустование | `/pad-clustering/{projectId}` | все |
| Параметры | `/parameters/*/{projectId}` | admin, analyst, viewer |
| Потоки | `/flows/technology/{projectId}` … | admin, analyst, viewer |
| Матрица | `/matrix/{projectId}` | admin, analyst, viewer |
| Отчёты | `/report/{projectId}` | admin, analyst, viewer |
| **Данные** | `/data/*/{projectId}` | подвкладки по роли (см. ниже) |
| Администрирование | `/admin/users`, `/admin/jobs`, `/admin/assistant` | admin |

**Данные** (`DataLayout`, подвкладки как у «Параметры»):

| Вкладка | Маршрут | Кто видит |
|---------|---------|-----------|
| Импорт | `/data/import/{projectId}` | admin, analyst, data_manager |
| Экспорт | `/data/export/{projectId}` | все роли |
| Импорт 3D | `/data/import-3d/{projectId}` | admin (загрузка); admin и владелец проекта (назначение, превью) |

Legacy URL `/import`, `/export`, `/import-3d` и `/{projectId}/data/...` → редирект на `/data/.../{projectId}`.

**Шапка приложения:** заголовок и subtitle текущей страницы — в `app-header` (`PageHeaderProvider`, `resolvePageHeader` + `stripProjectPrefix`); кнопки действий — в теле страницы.

**Администрирование:** вложенные вкладки (`AdminLayout`, как у «Параметры»): **Пользователи** (`/admin/users`), **Журнал задач** (`/admin/jobs`), **AI-помощник** (`/admin/assistant`). Прямой переход по URL поддерживается (GitHub Pages: `index.html` + `404.html`, service worker `public/sw.js`). **Выход** — иконка в нижней панели сайдбара (не в шапке).

Вкладки **Параметры:** пропускная способность (`/parameters/capacity/{projectId}`), **земляные работы** (`/parameters/earthwork/{projectId}` — L/W/H, опорная, поворот по точечным объектам), даты ввода (`/parameters/entry-dates/{projectId}`), **ставки** 16 показателей (`/parameters/rates/{projectId}`).

Вкладки **Логистика:** схема расчёта (`/logistics/schematic/{projectId}`), **объём песка** (`/logistics/sand/{projectId}`).

### 2.0 Режим 2.5D / 3D (просмотр на карте)

> Детали: [map-3d-features.md](../features/map/map-3d-features.md).

1. На `/map` доступен переключатель **2D | Площадки | 3D** (иконки на панели; 3D нужен `VITE_MAP_3D_ENABLED=true` и при необходимости `VITE_MAPTILER_KEY` в `frontend/.env`).
   При открытии вкладки «Карта» режим по умолчанию — **2D**.
2. Режим **«Площадки»** (контуры): на спутниковой подложке earthwork-eligible точечные объекты (включая **карьер песка**) показываются **полигонами площадок** (из схемы земляных работ или прямоугольник 120×80 м); **линии**, **POI** и **узлы** — без изменений. Рисование и редактирование доступны, как в **2D** (см. [pad-earthwork.md](../features/pad-earthwork/pad-earthwork.md) § «Режим „Площадки“ на карте»).
3. В **3D** доступны:
   - навигация (pitch/bearing), выбор объектов;
   - **панель «Слои»** (боковая колонка на `/map`): подложка (спутник, рельеф MapTiler, 3D-модели точек), группы объектов по подтипам, импортированные слои-источники, радиусы порогов;
   - **точки** — glTF-здания/трубы/резервуары (Kenney, CC0) или custom GLB; цвет слоя — для bundled-моделей;
   - **линии** — объёмные трубы по рельефу (дороги, нефть/газ/вода, ЛЭП и т.д.).
4. **Сохранение панели «Слои»:** включённые/выключенные чекбоксы (кроме «Источники» — они в БД проекта) запоминаются в браузере **для активного проекта** и восстанавливаются при возврате на «Карту» с других разделов (`/data/import-3d`, «Матрица», «Параметры» и т.д.). Ключ: `dm-map-layer-prefs:{projectId}`.
5. Рисование, линейка и перетаскивание геометрии — в **2D** и **Площадки**; в **3D** — только просмотр. **Групповое перемещение** — [§6.1.1](../features/map/map-objects-and-spatial-calculations.md). **Копирование/вставка** — [§6.1.0](../features/map/map-objects-and-spatial-calculations.md) и §2.0.1 ниже.
6. Вкладка «Дополнительно» у объекта:
   - `render_3d_height_m`, `render_3d_base_m`, **«Масштаб 3D (×)»** (`render_3d_scale`, 0.1–10);
   - **«Стиль 3D»** — glTF-модель или столбик (extrusion);
   - **«Модель 3D»** — стандартная Kenney по подтипу или custom GLB, назначенные на этот подтип на **Данные → Импорт 3D** (`/data/import-3d`).
7. Превью отчёта и матрица: live **3D**-карта; PNG-снимок отчёта по-прежнему **2D**.
8. Демо-данные для проверки: `python scripts/draw_demo_map_network.py` (из `backend/`).

### 2.0.1 Копирование и вставка группы (2D)

1. Включает **«Редактирование на карте»** → **«Выбор»** → **«Группа объектов»**.
2. Рамкой выделяет объекты для копии. Для цепочки **автодорог** обязательно включить **узлы/кусты на стыках** (и концевые опоры, если линия к ним привязана).
3. **Ctrl+C** или кнопка **«Копировать»** в панели группы.
4. **Ctrl+V** или **«Вставить»** → клик по карте в место вставки (центр группы переносится в эту точку).
5. Система создаёт копии с новыми именами; **длина и форма** каждой линии совпадают с оригиналом. Концы, чьи опоры были в выделении, остаются на копиях этих опор.
6. **Ctrl+X** — вырезать (копия + удаление с подтверждением). **Ctrl+Z** — отмена последней операции на карте.

Технические детали, поля API и ограничения — [map-objects-and-spatial-calculations.md](../features/map/map-objects-and-spatial-calculations.md) §6.1.0.

### 2.0.2 Соединение точечных объектов автодорогами *(interim)*

> **Целевой UX:** §2.0.4 «Построить сеть» и [autoroad-network-plan.md](../autoroad/autoroad-network-plan.md). Ниже — временный поток через групповое выделение.

1. **«Редактирование на карте»** → **«Выбор»** → **«Группа объектов»**; рамкой выделить **не менее двух** точечных объектов инфраструктуры (без линий и без POI).
2. В панели группы — **«Соединить автодорогами»** (доступна при праве записи инфраструктуры).
3. Система показывает сводку предварительного расчёта (`dry_run`): число новых линий, узлов, разрезов, суммарная длина, предупреждения (`far_from_autoroad`, `already_connected`, …). В **журнале задач** (шапка) — первый HTTP-шаг с `dry_run: true`.
4. После подтверждения — второй `POST .../infrastructure/autoroad-connect` (`dry_run: false`) или apply сети по JSON-конвейеру; при наличии сети — подъезд object→snap и связность **по сети**; без сети — 2 объекта: прямая `link`, 3+: **MST**; **≤1** `autoroad` на объект. Два POST при legacy connect — см. [task-log-panel.md](../features/jobs/task-log-panel.md) §5.
5. **Ctrl+Z** отменяет созданные этой операцией линии и узлы (не восстанавливает исходную геометрию разрезанных линий).

Алгоритм — [map-objects-and-spatial-calculations.md](../features/map/map-objects-and-spatial-calculations.md) §1.8. **Применение** — фоновая задача (§1.9).

### 2.0.4 Построить сеть автодорог

1. **«Редактирование на карте»** → инструмент **«Сеть»** (отдельно от группового выделения и legacy «Соединить автодорогами»).
2. В панели **«Сеть автодорог»** добавляет терминалы: **клик**, **рамка**, **«Видимые»** или **по типу** — любые точечные объекты инфраструктуры, кроме **Узел / Узел метанола / Узел ЛЭП** (`node`, `methanol_joint`, `power_line_node`). POI и линии не выбираются. Нужно **2–50** точек.
3. Параметры расчёта (солвер, лимит подъезда, роли start/end и др.) — в той же панели; настройки сохраняются в браузере.
4. Список терминалов: роли start/end/intermediate, поиск, удаление по одному, «Очистить».
5. **«Рассчитать»** — `request` → `compute` (в журнале задач — JSON-шаги); overlay на карте из `new_lines`, модалка с цифрами и предупреждениями.
6. **«Применить»** в модалке — `apply` с тем же `NetworkPlanResponse` (без повторного compute); при очереди — job `autoroad_connect`, опрос статуса; **Ctrl+Z** — как в §2.0.2. См. [autoroad-network-instruction.md](../autoroad/autoroad-network-instruction.md), [task-log-panel.md](../features/jobs/task-log-panel.md).

**Логика соединения:** результат **обязан быть связным** по всем выбранным терминалам. При `autoroad` — подъезд к ближайшей точке полилинии, пути по графу, мосты snap↔snap между компонентами. Без дорог — 2 `Т`: одна линия; 3+ `Т`: **MST** (минимальная суммарная длина), узлы на рёбрах, подъезды к сети. Терминал **не** перекрёсток; **≤1** линия на объект; `already_connected` — без второго подъезда. Подробности — [autoroad-network-plan.md](../autoroad/autoroad-network-plan.md), [map-objects §1.8](../features/map/map-objects-and-spatial-calculations.md).

### 2.0.3 Журнал задач (шапка)

На всех страницах с шапкой — иконка **слева от переключателя темы** (не путать с **Администрирование → Журнал задач** для админа).

1. Журнал использует **активный проект** из глобального store (`currentProjectId` — задаётся на страницах «Экспорт», «Карта», «Дашборд» и т.д.; селектора проекта в шапке нет).
2. Открывает панель журнала — список операций по **этому** проекту.
3. Видит:
   - **HTTP-цепочки** (например «Построение сети»: `request` / `compute` / `apply` с полными JSON);
   - **фоновые jobs** (импорт, логистика песка, apply в очереди) — `payload` и `result`.
4. Может **скачать JSON** одной записи или **экспорт всего** лога для отладки.
5. Для `pending` job — кнопка **«Отменить»** (`POST .../jobs/{id}/cancel`).

История HTTP хранится в **sessionStorage** вкладки (до ~30 записей); jobs подгружаются из `GET .../projects/{id}/jobs?limit=30`.

Подробно: [task-log-panel.md](../features/jobs/task-log-panel.md).

### 2.0.5 Фоновые расчёты (очередь)

Долгие операции (соединение автодорог **применить**, async-импорт, **логистика песка**, **анализ всех POI**) ставятся в очередь. Пока задача выполняется, второй запуск по тому же проекту отклоняется (**409**). UI показывает «выполняется…» и опрашивает статус до `completed` / `failed` (журнал в шапке и опрос `activeJob`).

Зависшие задачи (`pending` дольше ~15 мин или `running` дольше таймаута воркера) при новом запуске автоматически помечаются `failed`. Отмена вручную: `POST /projects/{id}/jobs/{job_id}/cancel` (для `pending` — сразу). Администратор: **Администрирование → Журнал задач** — глобальный список и `POST /admin/jobs/{id}/cancel` (см. §5.3).

Подробности: [map-objects-and-spatial-calculations.md](../features/map/map-objects-and-spatial-calculations.md) §1.9, [DEPLOY.md](../../DEPLOY.md) (Redis + worker).

### 2.1 Загрузка данных инфраструктуры (3 канала)

**Канал А: API корпоративных систем**
1. Переходит в **Данные → Импорт** (`/data/import`) в боковом меню (или из карточки проекта → «Импорт»)
2. В карточке **«Подключение API»** настраивает подключение
3. Выбирает тип реестра:
   - Реестр дорог (линейные объекты)
   - Реестр трубопроводов (нефть, вода, газ)
   - Реестр площадных объектов (ГКС, ПС, ГТЭС, водозабор)
   - Реестр энергообъектов (ЛЭП, подстанции)
4. Вводит URL API и параметры аутентификации (Bearer / API Key / Basic)
5. Тестирует подключение
6. Данные загружаются асинхронно и отображаются на карте слоями

**Канал Б: Импорт CSV**

Соответствие `type` ↔ геометрия — [map-objects-and-spatial-calculations.md](../features/map/map-objects-and-spatial-calculations.md) §1.4.

**Точечные** (`lat`, `lon`): `gas_processing`, `gtes`, `substation`, `refinery`.

**Линейные** (`start_lat`, `start_lon`, `end_lat`, `end_lon`): `autoroad`, `oil_pipeline`, `water_pipeline`, `power_line`.

1. Переходит в **Данные → Импорт** → карточка **«Импорт файлов»**
2. Загружает файл с колонками по типу подтипа (см. списки выше)
3. Система автоматически определяет категорию и подтип по колонке `type`; при несоответствии геометрии подтипу — ошибка (FR-2.5.4)
4. Пользователь проверяет классификацию и корректирует при необходимости
5. Подтверждает импорт

### 2.2 Импорт 3D (custom GLB)

1. **Администратор** или **владелец проекта** открывает **Данные → Импорт 3D** (`/data/import-3d`; вкладка видна при соответствующих правах)
2. Выбирает проект (если активный проект ещё не задан — через дашборд, карту или другой раздел с выбором проекта)
3. **Только администратор:** загружает файл `.glb` (до ~20 MB) в блоке «Загрузка GLB», указывает **высоту модели (м)**
4. **Владелец или администратор:** в блоке «Назначение подтипам» выбирает модель, отмечает **один или несколько** подтипов (чекбоксы); опционально включает **«Применить к объектам на карте»** (`только без модели` / `все объекты подтипа`) с превью счётчика → «Сохранить назначение»; «Снять все» очищает привязку
5. Справа (на широком экране) — **«Превью на карте»** (sticky при прокрутке форм и таблицы моделей); таблица «Модели проекта» — в левой колонке под формами
6. В таблице «Модели проекта» — имя, размер, высота, usage, подтипы; **«Изменить»** (PATCH имени/высоты); удаление с подтверждением, если модель используется
7. На **«Карте»** → карточка объекта → «Дополнительно» → **«Модель 3D»**: «Стандартная» или custom из списка для подтипа; сохранение пишет `render_3d_model_id = custom:{uuid}` для **конкретной** точки (или массово при assign с `apply_to_objects`)

**API (кратко):** `GET/POST /projects/{id}/map3d-custom-models`, `PATCH .../{model_id}`, `POST .../{model_id}/assign` с `{ "subtypes": ["gtes", "node"], "apply_to_objects": true, "apply_mode": "empty_only" }`, `GET .../apply-preview`, `GET .../file`, `DELETE`. Список: `display_name`, `file_size_bytes`, `usage_count`; assign — `{ model, objects_updated }`.

**Прод (GitHub Pages):** после входа клиент синхронизирует Bearer и CSRF (`syncClientAuthSession`); загрузка GLB на сервер и отображение в 3D/превью используют разные каналы — файл модели запрашивается с `Authorization: Bearer` (см. [auth-rbac.md](../architecture/auth-rbac.md), [map-3d-features.md](../features/map/map-3d-features.md) § custom GLB).

**Хранение на prod:** GLB на диске VM (`/opt/decision-matrix/shared/map3d_models`), bind-mount в контейнер — переживают redeploy. См. [map3d-models-storage.md](../deploy/map3d-models-storage.md).

**Если модели не видны после upload:** жёсткое обновление страницы (Ctrl+F5); проверить назначение подтипов и выбор «Модель 3D» на точке; при «404 (from disk cache)» — очистить кэш сайта; на VM — наличие файла в volume.

Линейные подтипы glTF не поддерживают. Подробности: [map-3d-features.md](../features/map/map-3d-features.md), wiki [import-3d](../wiki/articles/import-3d.md).

### 2.3 Экспорт данных проекта

> Детали: [project-export.md](../features/import-export/project-export.md).

1. Пользователь открывает **Данные → Экспорт** (`/data/export`) — вкладка в разделе «Данные».
2. На панели **«Проект для выгрузки»** выбирает проект (список всех доступных проектов; выбор сохраняется в `currentProjectId`).
3. На карточках форматов видит количество объектов для каждой выгрузки (точечные / все / точки и линии).
4. Скачивает нужный формат:
   - **Координаты точечных объектов** — Excel или CSV (подтипы `POINT_SUBTYPES`);
   - **GeoJSON проекта** — `.geojson` (совместим с импортом GeoJSON);
   - **Координаты всех объектов** — Excel или CSV (точки + линии с полным путём).
5. При пустом проекте — переход на «Карту» или «Импорт» из подсказки на странице.

**Канал В: Ручное нанесение**
1. Открывает раздел **«Карта»** (рисование только в **2D**)
2. Выбирает инструмент:
   - **«Точка»** — для `gas_processing`, `gtes`, `substation`, `refinery` и др. (меню подтипов)
   - **«Линия»** — для всех линейных подтипов (`autoroad`, `oil_pipeline`, `power_line`, `additional_line`, …)
3. **Линия:** первый клик — по точечному объекту на карте (иконка, точные координаты); далее — промежуточные вершины в любом месте; завершение — двойной ЛКМ/ПКМ, Enter или «Готово». Конец на объекте (точное совпадение координат / клик по иконке) или автоматическое создание **`node`** в пустом месте. Минимум 2 вершины.
4. **Точка:** клик по карте (при попадании на линию — вставка точки с разрезом линии).
5. Выбирает подтип, вводит название, сохраняет.
6. Координаты в БД — полная точность; в строке координат на карте и в полях карточки — **3 знака** после запятой (см. [map-objects-and-spatial-calculations.md](../features/map/map-objects-and-spatial-calculations.md) §1.5).

### 2.2 Визуализация пороговых радиусов
1. Пользователь открывает карту проекта
2. Вокруг точек интереса отображаются полупрозрачные круги (opacity 0.15) — пороговые радиусы:
   - **Серый круг** — допустимое расстояние до автодороги (например, 50 км)
   - **Бирюзовый круг** — допустимое расстояние до нефтепровода (40 км)
   - **Жёлтый круг** — допустимое расстояние до ЛЭП (30 км)
   - **Оранжевый круг** — допустимое расстояние до ГКС (80 км)
3. В панели слоёв пользователь может включить/выключить отображение каждого радиуса отдельно
4. Объекты инфраструктуры отображаются цветными иконками (Lucide/SVG) с белой рамкой и тенью

### 2.2.1 Производительность карты (крупные проекты)

При **≥ ~80** объектах в проекте:

1. **Просмотр** (кнопка «Редактирование на карте» **выключена**) — на слой попадают объекты в видимой области (`GET` с `bbox` и буфером ~12%). Пока вы слегка двигаете карту внутри уже загруженной зоны, **повторных запросов нет**; новый запрос — только при заметном сдвиге или зуме за пределы буфера.
2. **Редактирование** (**включено**) — всегда полный набор объектов в памяти: создание, удаление, вставка и перетаскивание сразу видны на карте без F5.
3. **Полный кэш** в фоне — для привязки концов линий, поиска и heal; после импорта или undo карта обновляется целиком (`invalidateMap`).

Созданный или изменённый объект в режиме просмотра остаётся на карте за счёт синхронизации кэшей (полный список + bbox-срез + кратковременный overlay до refetch). Подробности: [map-objects-and-spatial-calculations.md](../features/map/map-objects-and-spatial-calculations.md) §6.1.2.

**Объект метанола:** в меню **«Точка»** на панели карты (наряду с ГКС, НПЗ, узлом и др.); создание кликом по карте, вставка из буфера, выбор/перемещение, линии к опоре (точные координаты), поиск и слои — как у других точечных объектов; подтип в карточке **не меняется** после создания.

**Плавность 2D (просмотр и hover):** движение мыши не перерисовывает всю страницу — координаты курсора в React state обновляются только при вставке, рисовании точки/POI, линейке и черновике линии; hit-test и hover на карте — один раз за кадр (`requestAnimationFrame`), без чтения пикселей canvas. Подвал карты **не показывает** координаты курсора. Регрессию после релиза — [testing-strategy.md](../testing/testing-strategy.md) § «Карта 2D — ручной perf checklist».

### 2.3 Пространственный анализ
1. Выбирает точку интереса на карте
2. Нажимает "Анализировать окружение"
3. Система находит ближайший объект по каждому активному **внешнему** подтипу (точечные объекты: ГКС, ПС/ТП и т.д.)
4. Расстояние считается по прямой (geodesic) до **якоря**: для площадного объекта — до маркера; для линейного объекта (при ручном выборе) — до ближайшей точки на трассе, не обязательно до конца отрезка
5. На карте линия подключения идёт от POI к якорю (FR-10.3.1)
6. **Пример:** автодорога «М-11» хранится как `LineString`; при привязке линия от участка касается дороги в точке `ST_ClosestPoint`, а не только у начала полилинии
7. Результаты отображаются в боковой панели с расстояниями и статусами

### 2.4 Оптимизация размещения кустов с карты *(FR-5.3.5, ✅)*

> **«Кустование»** (`/pad-clustering`) — работа на **одном** кусте. **Этот поток** — **сколько и где** поставить **новые** кусты по забоям. **Число кустов в матрице POI** (FR-5.3.1) — только для стоимости, без координат.

1. На карте рисуют или импортируют **забои** (ННБ; для ГС — Т1 и Т3 = одна скважина).
2. В режиме **«Кусты»** / **«Оптимизация кустов»** выделяют нужные забои.
3. В панели задают ограничения: максимум скважин на куст, минимальное расстояние между кустами, при необходимости — проверку SF; в **«Расширенные»** — перебор центра по Σ MD ([input-parameters.md](input-parameters.md) §6).
4. **«Рассчитать»** — система перебирает варианты разбиения по кустам; для каждой группы — **двухфазный** перебор центра (грубая сетка → полный design), раскладка устьев и горизонтальный/connector design. При большом числе скважин — фоновая задача и [журнал задач](../features/jobs/task-log-panel.md) (таймаут до 10 мин).
5. На карте — **предпросмотр** кустов; в таблице — варианты от лучшего к худшему (число кустов, Σ MD, min SF, предупреждения).
6. **«Применить»** — создаются новые кусты, забои привязываются; **существующие кусты не меняются**.
7. Переход в **«Кустование»** (`/pad-clustering?padId=…`) — доработка раскладки, отметки устья и SF.

Подробнее: [pad-placement-optimization.md](../features/pad-placement/pad-placement-optimization.md), [инструкция](../wiki/articles/pad-placement-optimization.md).

### 3.1 Создание проекта
1. Пользователь нажимает "Новый проект" на Dashboard
2. Вводит название проекта (например, "Участок Западный")
3. Вводит описание (опционально)
4. Задаёт ставки стоимости в **Параметры → Ставки** (`/parameters/rates`, 16 показателей, **тыс. ₽**):
   - **Линейные внутренние (тыс. ₽/км)**: Автодорога, Нефтепровод, Водопровод, ЛЭП
   - **Площадные внешние (тыс. ₽ за объект)**: ГКС, ГТЭС, ПС/ТП, НПЗ
   - **Кустовые площадки (тыс. ₽/шт.)**
   - **Инженерное оборудование (тыс. ₽)**: ГТЭС/ГПЭС, локальная насосная, ГПЭС/ГТУ, подготовка нефти (МКОС/БМУПН/ЦПС/УПСВ)
5. В карточке проекта доступна краткая сводка ставок и ссылка «Открыть все ставки»
6. В разделе **«Пороги и нормы по умолчанию»** (`project_distance_defaults`, FR-4.1.5):
   - **Пороги до внешних объектов (4):** ГКС, ГТЭС, ПС/ТП, НПЗ — `max_distance_*_km`
   - **Макс. суммарная длина internal (4):** автодорога, нефтепровод, водопровод, ЛЭП — `max_total_line_*_km` (дефолты 50/40/30/30)
   - **Нормы км/КП (4):** по умолчанию 3.0 (FR-5.3.4)
7. Опционально публикует проект для Viewer (`visibility = published`, FR-1.2.6)
8. Сохраняет проект

### 3.2 Добавление точек интереса (POI)
1. Внутри проекта переходит на вкладку "Точки интереса" и нажимает **«Добавить на карте»** **или** на экране **«Карта»** включает инструмент **POI**
2. Кликает на карте в нужном месте — открывается **модальное окно** с координатами (только чтение) и полями исходных данных
3. Заполняет параметры в окне и нажимает **«Сохранить точку»**; маркер POI появляется на карте, данные синхронизируются с карточкой проекта
4. Альтернатива: правка уже созданной точки в форме проекта (координаты задаются только с карты)
5. Параметры точки:
   - **Флюид**: Нефть / Газ (`poi_fluid_type`) — влияет на активные подтипы (FR-5.1.5)
   - Название, плановый объём добычи (тыс. тн/год; подпись зависит от флюида)
   - **Объём закачки воды** (тыс. т/год), по умолчанию 0 — всегда в форме (FR-4.2.11)
   - **Добыча на 1 скважину**: 10 тыс. тн/год (по умолчанию)
   - **Скважин на одной кустовой площадке**: 4 (по умолчанию)
   - **Электроснабжение**: Внешнее / Внутреннее
   - **Способ закачки**: Централизованная / Локальная
   - **Утилизация газа**: В пласт / Факел / Электрогенерация
   - **Сбор продукции скважин**: Однотрубная / Двухтрубная / Комбинированная (FR-5.2.7)
   - **Подготовка нефти**: МКОС / БМУПН / ЦПС(УПН) / УПСВ / МФНС (без подготовки) (FR-5.2.4)
   - **Транспортировка**: Автовывоз / Морской порт / Магистральный трубопровод
6. Система автоматически рассчитывает (в модальном окне и в форме проекта):
   - Суммарное количество скважин = Объём / Добыча на скважину
   - Кустовые площадки = Math.ceil(Скважин / Скважин на КП)
7. В карточке POI — три блока (наследование из проекта, override, FR-4.2.6–4.2.13):
   - **«Пороги до внешних объектов (4)»**
   - **«Макс. суммарная длина internal (4)»**
   - **«Нормы линейной инфраструктуры (4)»** (км/КП)
8. Инженерные параметры (бейджи) задаются в карточке проекта; повторяет для нескольких POI

### 3.3 Анализ окружения
1. Выбирает точку интереса из списка
2. Нажимает "Анализировать окружение"
3. Система рассчитывает статус по **9 подтипам** (5 внутренних + 4 внешних) с учётом инженерных параметров; **поиск на карте** — только для активных **внешних** (FR-6.1.1):

   **Внутренние линейные (Inside POI)** — `distance_km = pads_count × km_per_pad` (FR-5.3.4), без поиска на карте:
   - Пример: 2 КП × 3 км/КП = 6 км автодороги; сравнение с `max_total_line_autoroad_km` (50 км) → `within_limit`
   - При `pads_count = 0`: `distance_km = 0`, `within_limit` если подтип активен (FR-6.2.1b)
   - Нефтепровод / Водопровод / ЛЭП — по той же формуле, если подтип активен по `eng_*`
   - **Кустовые площадки**: 2 шт. (FR-5.3.1)

   **Внешние параметры (Outside POI)** — ищутся на карте:
   - ГКС: ближайший в 92 км (превышает лимит 80 км) ⚠️
   - ГТЭС: не требуется (электроснабжение — "Внешнее") —
   - ПС / ТП: ближайший в 22 км (в пределах лимита 25 км) ✅
   - НПЗ: не требуется (транспортировка — "Автовывоз") —

4. Результаты отображаются в таблице с группировкой
5. **Инженер может выбрать альтернативный объект** для внешнего подтипа из списка кандидатов (FR-6.3.4)
6. Результаты сохраняются в `poi_infrastructure_analysis` (при повторном анализе перезаписываются)
7. На карте отображается точка интереса и найденные внешние объекты

### 3.4 Базовый расчёт вариантов
1. Нажимает "Рассчитать базовый вариант"
2. Система использует сохранённые результаты анализа
3. Система применяет инженерные параметры точки:
   - **Электроснабжение**: "Внешнее" → ЛЭП активен, ГТЭС/ГПЭС исключены; "Внутреннее" → ГТЭС/ГПЭС активны, ЛЭП исключён
   - **Закачка**: влияет на внутренний **Водопровод** (FR-5.2.2, FR-5.2.6)
   - **Газ**: влияет на ГТЭС/ГПЭС (FR-5.2.3)
   - **Подготовка нефти**: стоимость оборудования по выбранному типу
   - **Транспортировка**: "Автовывоз" → Нефтепровод, НПЗ исключены; "Морской порт" → Нефтепровод активен (без отдельного внешнего подтипа порта); "Магистральный" → Нефтепровод и НПЗ активны
4. **Система создаёт один базовый вариант** (`implementation_variants` + записи в `variant_infrastructure_items` по каждому активному подтипу)
5. Для варианта рассчитывается:
   - **Линейные внутренние**: `pads_count × km_per_pad` → расстояние × ставка ₽/км (FR-7.3.1)
   - **Кустовые площадки**: количество × ставка за шт.
   - **Площадные внешние**: фиксированная стоимость за объект (НПЗ при магистральной транспортировке; при «Морской порт» — только нефтепровод)
   - **Инженерное оборудование** по 5 параметрам
   - Общая стоимость = сумма всех активных подтипов + сумма инженерного оборудования
   - Общий статус: `exceeds_limit` > `construction_required` > `within_limit`; `not_required` не учитывается
6. Базовый вариант отображается в таблице

### 3.5 Матрица решений для точки
1. Все точки интереса отображаются в **вертикальной матрице** (режим «Таблица» по умолчанию): строки = параметры, столбцы = POI; опционально — режим **«Карточки»**
2. **Параметры разделены на визуальные группы** (внутренние, инженерные бейджи, инженерное оборудование, внешние):

   **Внутренние решения (Inside POI)**:
   - Автодорога, Нефтепровод, Водопровод, ЛЭП, Кустовые площадки
   - Сбор скважин, Электроснабжение, Закачка, Утилизация газа, Подготовка нефти

   **Внешние объекты (Outside POI)**:
   - ГКС, ГТЭС, ПС/ТП, НПЗ
   - Транспортировка

   **Инженерное оборудование** (детализация по 5 параметрам)

3. **Инженерные параметры — группы бейджей**: все значения серым, выбранное подсвечено цветом
4. **Статусы подтипов**: ✅ В норме / ⚠️ Превышение / 🔧 Строительство / — Не требуется
5. Инженер может:
   - Изменить инженерный параметр → система пересчитывает связанные подтипы
   - Отфильтровать POI с превышениями
   - Перейти к формированию отчёта по выбранной точке

### 3.6 Визуализация на карте
1. Инженер выбирает точку интереса в матрице или на карте
2. На карте отображаются:
   - Точка интереса (выделена)
   - Линии подключения к объектам (цвет = статус)
   - Объекты инфраструктуры (цветные иконки)
3. При переключении между POI линии обновляются

### 3.7 Экспорт одностраничника для руководства (FR-11)

1. Инженер открывает раздел **«Отчёты»** (`/report`) — список сохранённых одностраничников проекта.
2. Нажимает **«Подготовить отчёт»** (`/report/new`), выбирает **POI**.
3. Система строит превью одностраничника:
   - **Заголовок**: название участка, координаты, инженер, дата
   - **Карта**: точка + линии до объектов (live-анализ POI)
   - **Таблица**: 9 подтипов с группировкой Внутренние/Внешние + стоимость
   - **Инженерные параметры**: бейджи + стоимость оборудования
   - **Дорожная карта**: 6 этапов (редактируемые сроки)
   - **Рекомендация**: шаблонный текст + редактирование
4. **Сохранение** — `POST /api/v1/projects/:id/one-pagers`; просмотр/редактирование — `/report/:id`.
5. **Экспорт**:
   - **PDF** — клиентская печать (`window.print()` + `@media print`), кнопка в превью и списке
   - **PPTX** — `POST .../one-pagers/:id/export/pptx` (backend `python-pptx`, карта из `map_snapshot_base64`)
6. В списке отображаются статусы генерации PPTX: `pending` | `ready` | `error`.
7. Роль **viewer** — read-only: список, превью, PDF; без сохранения и PPTX.

### 3.9 Схема потоков (PFD)

> Подробная спецификация: [fluid-flow-schematic.md](../features/flows/fluid-flow-schematic.md).

1. Инженер открывает раздел **«Потоки»** (`/flows`) в боковом меню.
2. Выбирает **точку интереса** в списке.
3. Система строит схему: **POI → Сепарация →** ветки **Нефть / Вода / Газ** по правилам `eng_*` и **графу сети из БД** (топология после `build_network_from_lines`; на карте не отображается).
4. **Вода (нефтяной POI):**
   - ветка **всегда** присутствует;
   - **локальная** закачка → **«В пласт»**;
   - **централизованная** → **БКНС** → **«В пласт»**;
   - **объём закачки** (`water_injection_volume`) — во всплывающем окне блока **«В пласт»**.
5. Инженер может **редактировать** макет (перемещение, связи, лимиты), **сохранить** layout или **пересчитать** расчётную схему.
6. На карте для точечных объектов (кроме исключений) задаётся **пропускная способность** — используется на терминалах авто-схемы и для проверки перегрузки (красная подсветка).

### 3.10 Логистика песка

1. На **карте** размещают карьеры (`sand_quarry`) с **начальным** и **текущим** объёмом, потребителей — с **объёмом спроса** (`sand_volume_m3`) и/или **планом по годам** (`sand_volume_by_year` в карточке → Логистика), линии **автодорог** и узлы **`node`**. Даты ввода — **Параметры → Дата ввода** или карточка объекта.
2. Backend пересобирает сеть из линий (при создании линий, импорте; перед расчётом логистики — автоматически). **Узлы и рёбра графа на карте не показываются.**
3. Открывают **Логистика → Логистика** (`/logistics/schematic`), задают **горизонт расчёта** (начало — автоматически, конец — авто с возможностью правки) и **срез для схемы** (`as_of` при расчёте), нажимают **«Рассчитать логистику песка»**.
4. Система симулирует распределение **по календарным годам** внутри горизонта: на каждый год — граф `autoroad`, введённые к 31.12.Y объекты, **глобальное** начисление спроса, **жадное** распределение неудовлетворённого остатка (backlog) с depletion карьеров; сохраняет `timeline[]` и снимок `subnets` на выбранный `as_of`. Потребитель с ранним вводом (например 2019) **ждёт** появления песка и не выбывает из расчёта — отгрузка в первый год с доступным карьером/путём. Пропорциональная альтернатива считается на накопительный спрос среза.
5. **Срез на схеме:** после расчёта можно выбрать другой год (временная шкала с чипами) — таблицы берут объёмы из `timeline`, схема показывает **все** объекты подсети (будущие — серые), **без пересчёта** и без сброса камеры. Баннер при расхождении выбранного среза и сохранённого `as_of`.
6. Объекты без пути к карьеру по автодорогам попадают в предупреждения; на схеме канонической подсети они могут отображаться серыми, если есть snap и координаты.
7. После расчёта таблица **«Плечо возки»** (карьер, объём, км по пропорциональному распределению) доступна для **всех точечных объектов с полем спроса** (`sand_volume_m3`, кроме узла и карьера): вкладка **Логистика** карточки на карте, столбец на **Логистика → Объём песка**, колонка в таблице потребителей на **Логистика → Логистика**. Колонка **«По годам (отгрузка)»** — из `allocation_by_year_m3`.
8. Результат **сохраняется в БД** проекта (`project_sand_logistics_results`, поля `horizon_from`, `horizon_to`, `timeline` в JSON); доступен после перезагрузки вкладки и на других устройствах. `POST .../analyze` — пересчёт и сохранение. При **первом** открытии логистики в вкладке выполняется один `GET .../sand-logistics/result`; при переходах по приложению (карта, параметры, другие вкладки «Потоки») тот же расчёт берётся из памяти и `sessionStorage` вкладки **без повторного запроса**, пока пользователь не пересчитает или не обновит проект. Срез схемы, активная подсеть и настройки отображения по-прежнему хранятся в `sessionStorage` отдельно.
9. **Схема движения песка** в каждой подсети: временная шкала, геопривязка по lon/lat, адаптивные зазоры между блоками; на любом срезе видны все объекты (будущие — серые), цвет активных — по факту отгрузки, плановые пунктирные плечи. Подробнее — [map-objects-and-spatial-calculations.md](../features/map/map-objects-and-spatial-calculations.md) §1.7.1.
10. Остаток на карьере после отгрузки обновляют вручную в карточке объекта; симуляция не синхронизирует ручные правки между годами автоматически.

## 5. Администрирование

### 5.1 Управление пользователями (Admin, FR-1.4)
1. Переходит в **Администрирование → Пользователи** (FR-12.2.2)
2. Видит список всех пользователей
3. Может изменить роль пользователя
4. Может деактивировать аккаунт

### 5.2 Статистика на вкладке пользователей (Admin)
1. На вкладке **Пользователи** видит счётчики: пользователи, проекты, POI (из `/admin/stats`)

### 5.3 Журнал фоновых задач (Admin)
1. Переходит в **Администрирование → Журнал задач** (`AdminJobsPage`)
2. Видит панель здоровья очереди: доступность Redis, имя очереди, счётчики задач по статусам (`pending`, `running`, `completed`, `failed`, `cancelled`)
3. Пока есть активные задачи (`pending` / `running`), журнал и счётчики **автообновляются каждые 3 с** (подпись «Автообновление»); кнопка **Обновить** — ручной сброс
4. Фильтрует список по статусу, типу задачи или ID проекта (при смене фильтра страница сбрасывается на первую)
5. В таблице видит **10 последних записей на страницу** (сортировка по дате создания, новые сверху); счётчик «Показано N–M из K»; при K > 10 — **Назад** / **Вперёд**
6. В таблице видит проект (ссылка), инициатора, тип, статус, длительность и текст ошибки
7. Для задач **`pending` / `running`** в колонке «Действия» отображается **Отменить** (`POST /admin/jobs/{id}/cancel`); для завершённых, с ошибкой и отменённых — кнопки нет (прочерк)
8. Если задача уже завершилась до отмены, API возвращает актуальный статус (`completed` / `failed` / `cancelled`); UI показывает соответствующее уведомление без ошибки

### 5.4 Параметры LLM (Admin)
1. Переходит в **Администрирование → AI-помощник** (`AdminAssistantPage`, `/admin/assistant`)
2. На вкладке **Статус и настройка** видит карточки: LLM-провайдер (probe), веб-чат, Wiki RAG, runtime override; при загрузке выполняется автоматическая проверка подключения
3. Раскрывает **Детали проверки** — HTTP-коды для chat и embeddings; при проблемах с embeddings — **Как исправить** (модалка + подстановка `nomic-embed-text`)
4. В **Текущей конфигурации** — effective `base_url`, модель, max tokens, timeout, embedding; API key только маской; отличия от `.env` подсвечены
5. В **Временном override** (подвкладки Chat / Wiki RAG): задаёт параметры, при необходимости выбирает модель из списка провайдера или вводит вручную, **Тестовый запрос**, **Применить** (`POST /admin/assistant/llm-config`) — чат сразу использует новые параметры без рестарта backend
6. **Сбросить override** (`DELETE /admin/assistant/llm-config`) возвращает значения из `app.env` / `.env`
7. На вкладке **Справка и пресеты** — карточки провайдеров (**Подставить**), локальные пресеты браузера, настройка Cursor MCP
8. Постоянные изменения на VM — `ASSISTANT_LLM_*`, `ASSISTANT_WIKI_EMBEDDING_*` в `app.env` и перезапуск контейнера API ([DEPLOY.md](../../DEPLOY.md))
