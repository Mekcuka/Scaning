## 1. Управление пользователями и доступом

### 1.1 Регистрация и аутентификация
- **FR-1.1.1**: Пользователь может зарегистрироваться по email и паролю
- **FR-1.1.2**: Пользователь может войти в систему по email и паролю
- **FR-1.1.3**: Система поддерживает JWT-токены с автоматическим обновлением
- **FR-1.1.4**: Пользователь может выйти из системы (инвалидация токена)
- **FR-1.1.5**: Пароль должен содержать минимум 8 символов, включая цифру и букву

### 1.2 Ролевая модель
- **FR-1.2.1**: Система поддерживает 4 роли: Admin, Analyst, Data Manager, Viewer
- **FR-1.2.2**: Admin может управлять пользователями и их правами
- **FR-1.2.3**: Analyst может создавать проекты, точки интереса и отчёты
- **FR-1.2.4**: Data Manager может загружать и управлять слоями инфраструктуры на карте
- **FR-1.2.5**: Viewer может только просматривать опубликованные проекты и отчёты
- **FR-1.2.6**: Проект имеет видимость `private` (по умолчанию, только владелец и роли с правом редактирования) или `published` (доступен Viewer на чтение). MVP: без multi-tenant — все опубликованные проекты видны любому Viewer

### 1.4 Администрирование
- **FR-1.4.1**: Admin может просматривать список пользователей, назначать роль, деактивировать учётную запись
- **FR-1.4.2** *(Should Have)*: Admin видит агрегированную статистику (число пользователей, проектов, POI)

### 1.3 Профиль пользователя
- **FR-1.3.1**: Пользователь может изменить имя, аватар и контактные данные
- **FR-1.3.2**: Пользователь может просмотреть историю своих действий
- **FR-1.3.3** *(Should Have)*: Система записывает в `audit_log` операции create/update/delete для проектов, POI и импорта

## 2. Картографический модуль

### 2.1 Базовая карта
- **FR-2.1.1**: Пользователь может просматривать интерактивную карту с масштабированием
- **FR-2.1.2**: Доступны базовые слои: OpenStreetMap, Satellite, Terrain  
  - **Реализация (май 2026):** 2D — подложка **спутник** (Esri World Imagery), вкл/выкл в панели слоёв; **OSM** как отдельный переключатель 2D — не реализован. **Terrain** — режим **3D** (MapTiler DEM, `VITE_MAPTILER_KEY`), см. [map-3d-features.md](../features/map-3d-features.md). Целевое соответствие FR — post-MVP для полного набора 2D-баз.
- **FR-2.1.3**: Карта поддерживает масштаб от уровня страны до уровня здания
- **FR-2.1.4**: Система отображает координаты курсора в реальном времени (в UI — 3 знака после запятой; в БД при сохранении с карты — полная точность `float`, см. `coords.ts`)

### 2.2 Слои данных
- **FR-2.2.1**: Пользователь может создавать пользовательские слои инфраструктуры
- **FR-2.2.2**: Каждый слой имеет название, тип (vector/raster), цвет и прозрачность
- **FR-2.2.3**: Пользователь может включать/выключать видимость слоёв
- **FR-2.2.4**: Пользователь может менять порядок наложения слоёв
- **FR-2.2.5**: Слои сохраняются между сессиями

### 2.3 Геообъекты инфраструктуры
- **FR-2.3.1**: Пользователь может добавлять точки (площадные объекты, подстанции) и линии (дороги, трубопроводы, ЛЭП) на карту
- **FR-2.3.2**: Каждый объект имеет название, описание, категорию, подтип и набор свойств (JSON)
- **FR-2.3.3**: Пользователь может редактировать геометрию объектов
- **FR-2.3.4**: Пользователь может удалять объекты
- **FR-2.3.5**: Объекты можно выбирать для просмотра деталей
- **FR-2.3.6**: Пользователь может искать объекты по названию и свойствам
- **FR-2.3.7**: Геометрия объекта согласована с подтипом: внешние площадные и подстанции — `POINT`; линейные подтипы (`autoroad`, `oil_pipeline`, `water_pipeline`, `power_line`) — `LINESTRING` или `MULTILINESTRING` (см. [map-objects-and-spatial-calculations.md](../features/map-objects-and-spatial-calculations.md) §1.4)
- **FR-2.3.8**: POI хранится как отдельная сущность `points_of_interest` с геометрией `POINT`; не смешивается со слоями инфраструктуры
- **FR-2.3.9**: Распределение подтипов по геометрии (жёсткое, 1:1): **точечные** — `gas_processing`, `gtes`, `substation`, `refinery`; **линейные** — `autoroad`, `oil_pipeline`, `water_pipeline`, `power_line`. Polygon/MultiPolygon для площадных объектов в MVP не поддерживается

### 2.4 Пространственный анализ
- **FR-2.4.1**: Система может находить ближайший объект заданного **внешнего** подтипа (Point) от точки интереса среди объектов проекта. Для **внутренних линейных** подтипов автопоиск в MVP не выполняется — длина по FR-5.3.4. Geodesic до LineString — только post-MVP / визуализация (см. [calculation-functions.md](../calculations/calculation-functions.md) §1, §3)
- **FR-2.4.2**: Система рассчитывает расстояние по прямой (geodesic, PostGIS `geography`) от POI до **якоря** объекта: для `POINT` — до координат объекта; для `LINESTRING`/`MULTILINESTRING` — до ближайшей точки на линии (`ST_ClosestPoint`)
- **FR-2.4.4**: Результаты анализа сохраняют `anchor_type` (`point_object` | `line_nearest_point`) и `anchor_geometry` (`POINT`) для отрисовки линий подключения и повторного расчёта
- **FR-2.4.5** *(planned)*: Поддержка явного графа инфраструктуры (узлы/рёбра); расстояние по прямой до вершины графа (`anchor_type = network_node`); топология и DDL — см. [map-objects-and-spatial-calculations.md](../features/map-objects-and-spatial-calculations.md) §5

### 2.5 Загрузка инфраструктуры
- **FR-2.5.1**: Пользователь может подключить внешний REST API корпоративных систем для загрузки инфраструктуры (реестры дорог, трубопроводов, площадных и энергообъектов). Поддержка аутентификации: Bearer, API Key, Basic
- **FR-2.5.2**: Данные загружаются автоматически и отображаются на карте слоями
- **FR-2.5.3**: Пользователь может импортировать файлы GeoJSON, CSV, Shapefile (.shp, .shx, .dbf), KML/KMZ. При импорте CSV поддерживаются колонки: `name`, `type`, `lat`, `lon` (точечные) или `start_lat`, `start_lon`, `end_lat`, `end_lon` (линейные)
- **FR-2.5.4**: Система автоматически определяет категорию и подтип по колонке `type`, валидирует данные и показывает ошибки; **геометрия строки должна соответствовать подтипу** (FR-2.3.9): точечные подтипы — только `lat`/`lon` или GeoJSON Point; линейные — `start_*`/`end_*` или LineString; несоответствие отклоняется с сообщением об ошибке
- **FR-2.5.5**: Импорт выполняется асинхронно с отображением прогресса
- **FR-2.5.6**: Пользователь может кликать на карте для размещения точечных объектов и проводить линии для линейных объектов (раздел «Карта»)
- **FR-2.5.7**: При ручном размещении открывается диалог с выбором категории, подтипа и названия
- **FR-2.5.8**: UI загрузки API и файлов вынесен в **отдельный раздел навигации «Импорт»** (не вкладка экрана «Карта»); отображается история последних импортов
- **FR-2.5.9**: Учётные данные API-коннекторов хранятся в `import_connections` в зашифрованном виде; не попадают в `import_logs` и ответы API
- **FR-2.5.10**: Каждая операция импорта фиксируется в `import_logs`: статус, счётчики записей, список ошибок (FR-2.5.5)
- **FR-2.5.11** *(реализовано, июнь 2026)*: **Данные → Экспорт** (`/data/export`) — выбор проекта в панели; три карточки форматов с количествами объектов; выгрузка координат точечных объектов (Excel/CSV), координат всех объектов (Excel/CSV), GeoJSON проекта (`.geojson`, совместим с FR-2.5.3). Клиентская реализация — [project-export.md](../features/project-export.md)

## 4. Управление проектами

> Нумерация раздела **4** соответствует префиксу требований **FR-4.x** (исторически раздел в документе шёл под номером 3).

### 4.1 Создание проекта
- **FR-4.1.1**: Пользователь может создать проект с названием и описанием
- **FR-4.1.2**: В настройках проекта задаются ставки стоимости строительства (**16 показателей**, ввод в **тыс. ₽**; в UI — вкладка **Параметры → Ставки** `/parameters/rates`; в карточке проекта — краткая сводка):
  - **Линейные внутренние (тыс. ₽/км)**: Автодорога, Нефтепровод, Водопровод, ЛЭП — 4 показателя
  - **Площадные внешние (тыс. ₽ за объект)**: ГКС, ГТЭС, ПС/ТП, НПЗ — 4 показателя
  - **Кустовые площадки (тыс. ₽/шт.)** — 1 показатель
  - **Инженерное оборудование (тыс. ₽, фиксированные ставки)** — 4 показателя:
    - Электроснабжение — Внутреннее (ГТЭС/ГПЭС)
    - Закачка — Локальная (автономная насосная)
    - Утилизация газа — Электрогенерация (ГПЭС/ГТУ)
    - Подготовка нефти — МКОС, БМУПН, ЦПС(УПН), УПСВ (4 ставки; МФНС = 0)
    - Транспортировка — все варианты = 0
- **FR-4.1.3**: Пользователь может редактировать параметры проекта (включая ставки и пороговые расстояния по умолчанию)
- **FR-4.1.4**: Пользователь может удалить проект
- **FR-4.1.5**: На уровне проекта в `project_distance_defaults` задаются: **(а)** пороги расстояния до **внешних** Point-объектов (4 подтипа: ГКС, ГТЭС/ГПЭС, ПС/ТП, НПЗ), км; **(б)** **макс. суммарная длина** internal linear (4 подтипа: автодорога, нефтепровод, водопровод, ЛЭП), км — `max_total_line_*_km` (дефолты 50/40/30/30); **(в)** нормы **км/КП** (4 подтипа, дефолт 3.0). При создании POI значения копируются; изменение дефолтов проекта не пересчитывает уже созданные POI

### 4.2 Точки интереса (POI)
- **FR-4.2.1**: В один проект можно добавить несколько точек интереса
- **FR-4.2.2**: Точка интереса размещается на карте (клик или ввод координат)
- **FR-4.2.3**: Для точки задаётся: название, плановый объём добычи (тыс. тн/год), описание
- **FR-4.2.4**: Для точки задаются параметры для расчёта кустовых площадок:
  - Добыча на 1 скважину (тыс. тн/год) — по умолчанию 10
  - Скважин на одной кустовой площадке — по умолчанию 4
- **FR-4.2.5**: Количество кустовых площадок рассчитывается автоматически:
  - Суммарное количество скважин = Объём добычи / Добыча на 1 скважину
  - Количество кустовых площадок = Суммарное количество скважин / Скважин на одной КП
- **FR-4.2.6**: Каждая точка интереса имеет **индивидуальные** пороги: 4 внешних (`max_distance_*`) и 4 макс. длины internal (`max_total_line_*`), переопределение относительно дефолтов проекта (FR-4.1.5)
- **FR-4.2.7**: При создании POI пороги и нормы **наследуются** из `project_distance_defaults`; инженер может изменить любое значение в форме точки
- **FR-4.2.8**: Единица планового объёма добычи в MVP — **тыс. тн/год** (`production_unit = thousand_tons_per_year`); поля `production_per_well` и `wells_per_pad` обязательны для расчёта кустовых площадок
- **FR-4.2.9**: В форме POI — аккордеон **«Пороги до внешних объектов (4)»** (`threshold_gas_processing_km`, `threshold_gtes_km`, `threshold_substation_km`, `threshold_refinery_km`)
- **FR-4.2.12**: Блок **«Нормы линейной инфраструктуры (4)»** — `km_per_pad_*` (FR-5.3.4)
- **FR-4.2.13**: Блок **«Макс. суммарная длина internal (4)»** — `max_total_line_*_km`; сравнение с `pads_count × km_per_pad` (FR-6.2)
- **FR-4.2.10**: Для точки интереса задаётся **тип флюида**: Нефть (`oil`) / Газ (`gas`); id `poi_fluid_type`. Влияет на расчёт доступности подтипов (FR-5.1.5). Подпись поля объёма добычи в UI зависит от флюида
- **FR-4.2.11**: Для точки задаётся **объём закачки воды** (тыс. т/год), id `poi_water_injection_volume`; отдельно от объёма добычи (FR-4.2.3). Поле **всегда** отображается в форме POI; агент закачки в MVP — только **вода**

## 5. Инженерные параметры точки интереса

### 5.1 Общие положения
- **FR-5.1.1**: Параметры точки интереса разделены на две группы: **Внутренние (Inside POI)** и **Внешние (Outside POI)**
- **FR-5.1.2**: Внутренние параметры — расчитываются и применяются внутри точки интереса
- **FR-5.1.3**: Внешние параметры — находятся в окружении точки интереса
- **FR-5.1.4**: Изменение инженерного параметра автоматически пересчитывает доступность связанных подтипов
- **FR-5.1.5**: **Тип флюида** (`poi_fluid_type`) применяется до инженерных правил FR-5.2:
  - **Нефть (`oil`)**: действуют правила FR-5.2–5.4 без изменений
  - **Газ (`gas`)**: подтип **Нефтепровод** — `not_required`, если транспортировка = «Автовывоз»; при «Морской порт» или «Магистральный трубопровод» — действуют правила транспортировки FR-5.4.1. Стоимость **подготовки нефти** = 0. Подтип **НПЗ** — `not_required`

### 5.2 Внутренние инженерные параметры (взаимоисключающие бейджи)
- **FR-5.2.1**: **Электроснабжение**: Внешнее / Внутреннее
  - Внешнее → ЛЭП активен, ГТЭС/ГПЭС исключены
  - Внутреннее → ЛЭП исключён, ГТЭС/ГПЭС активен
- **FR-5.2.2**: **Способ закачки**: Централизованная / Локальная
  - Централизованная → внутренний **Водопровод** по правилам Inside POI
  - Локальная → при `poi_water_injection_volume` > 0 активен внутренний **Водопровод** (FR-5.2.6)
- **FR-5.2.3**: **Утилизация газа**: В пласт / Факел / Электрогенерация
  - В пласт / Факел → ГТЭС/ГПЭС неактивны
  - Электрогенерация → ГТЭС/ГПЭС активны только при «Внутреннее» электроснабжении (при «Внешнее» — неактивны)
- **FR-5.2.4**: **Подготовка нефти** (`eng_oil_preparation`): один из вариантов — **МКОС**, **БМУПН**, **ЦПС(УПН)**, **УПСВ**, **МФНС (без подготовки)**. Стоимость инженерного оборудования (FR-7.3.4): МФНС = 0; иначе фиксированная ставка по типу из FR-4.1.2
- **FR-5.2.7**: **Сбор продукции скважин** (`eng_well_gathering`): **Однотрубная** / **Двухтрубная** / **Комбинированная**; отображается в блоке **внутренних решений** матрицы; в MVP **не влияет** на расчёт стоимости и статусы подтипов
- **FR-5.2.6**: **Объём закачки воды** (`poi_water_injection_volume`, тыс. т/год): агент — только вода; поле всегда в форме POI. При 0 закачки — по FR-5.2.2. При > 0 и **локальной** закачке — активен внутренний **Водопровод**; при **централизованной** — по FR-5.2.2. Значение показывается в матрице и одностраничнике; масштабирование стоимости от объёма в MVP не выполняется

### 5.3 Расчёт кустовых площадок (Inside POI)
- **FR-5.3.1**: Система автоматически рассчитывает количество кустовых площадок:
  - Суммарное количество скважин = Объём добычи (тыс. тн/год) / Добыча на 1 скважину (тыс. тн/год)
  - Количество кустовых площадок (шт.) = Math.ceil(Суммарное количество скважин / Скважин на одной КП) — округление всегда вверх
- **FR-5.3.2**: Результат отображается в матрице как вычисляемое значение (без расстояния и статуса)
- **FR-5.3.3**: Инженер может вручную скорректировать добычу на 1 скважину и скважин на КП в параметрах точки
- **FR-5.3.4**: Для каждого **активного внутреннего линейного** подтипа расчётная длина в базовом варианте: `distance_km = pads_count × km_per_pad(subtype)` (см. [calculation-functions.md](../calculations/calculation-functions.md) §3). `pads_count` — FR-5.3.1. `km_per_pad` — отдельно для автодороги, нефтепровода, водопровода, ЛЭП; дефолт на проекте (FR-4.1.5), переопределение на POI (FR-4.2.12). В матрице отображается подпись вида «{km_per_pad} км/КП × {pads_count} КП = {distance_km} км»
- **FR-5.3.5** *(Could Have, ✅ реализовано)*: **Оптимизация размещения кустов** — по выбранным на карте забоям система подбирает **сколько и где** поставить **новые** нефтяные/газовые площадки, стремясь к минимуму кустов и минимальной сумме **MD**; уже существующие кусты не изменяются. **Не заменяет** FR-5.3.1 (число КП в матрице POI без координат). Спецификация: [pad-placement-optimization.md](../features/pad-placement-optimization.md)

### 5.4 Внешний инженерный параметр
- **FR-5.4.1**: **Транспортировка продукции**: Автовывоз / Морской порт / Магистральный трубопровод
  - Автовывоз → Нефтепровод исключается, НПЗ неактивен
  - Морской порт → Нефтепровод активен (стоимость по ставке нефтепровода; отдельный внешний подтип «Морской порт» не используется), НПЗ неактивен
  - Магистральный трубопровод → Нефтепровод активен (строится до найденного НПЗ), НПЗ активен

## 6. Анализ окружения (9 подтипов)

### 6.1 Классификация подтипов
- **FR-6.1.1**: Система учитывает **9 подтипов** инфраструктуры (4 внутренних линейных + 4 внешних + кустовые площадки). **Поиск ближайшего объекта на карте** — только для **активных внешних** Point-подтипов (ГКС, ГТЭС/ГПЭС, ПС/ТП, НПЗ). **Внутренние линейные** (автодорога, нефтепровод, водопровод, ЛЭП): `distance_km` по FR-5.3.4, без привязки к объекту на карте в базовом варианте. LineString на карте — импорт/отображение (FR-2.3), не источник длины в MVP
- **FR-6.1.2**: Система автоматически находит ближайший Point-объект по каждому активному **внешнему** подтипу
- **FR-6.1.3**: Для каждого активного **внешнего** подтипа рассчитывается geodesic-расстояние от POI до якоря `point_object` (координаты объекта). Для активных **внутренних линейных** — `distance_km` по FR-5.3.4, `distance_source = pads_per_pad_formula`, якорь на карте не строится

### 6.2 Статусы подтипов
- **FR-6.2.1**: Каждый подтип имеет один из статусов (приоритет при агрегации: `exceeds_limit` → `construction_required` → `within_limit` → `not_required`):
  - `exceeds_limit` — превышение лимита (красный)
  - `construction_required` — строительство собственного (янтарный)
  - `within_limit` — в пределах лимита (зелёный)
  - `not_required` — не требуется (серый)
- **FR-6.2.1a**: **Внешние Point:** `exceeds_limit` если geodesic `distance_km` > `max_distance_*_km`; `construction_required` если объект не найден или FR-6.3.2
- **FR-6.2.1b**: **Внутренние линейные:** `exceeds_limit` если `pads_count × km_per_pad` > `max_total_line_*_km`; `construction_required` только FR-6.3.2; при `pads_count = 0` — `distance_km = 0`, `within_limit` если подтип активен
- **FR-6.2.2**: Подтипы со статусом `not_required` имеют стоимость = 0 принудительно и не участвуют в определении общего статуса
- **FR-6.2.3**: Общий статус варианта определяется по наихудшему приоритету среди активных подтипов: если есть `exceeds_limit` → общий статус `exceeds_limit`, если нет но есть `construction_required` → общий статус `construction_required`, и т.д.
- **FR-6.2.4**: Вариант помечается красным, если хотя бы один активный подтип имеет `exceeds_limit`

### 6.3 Ручное переопределение
- **FR-6.3.1**: Для **внешних** подтипов инженер может выбрать альтернативный Point-объект из списка кандидатов (FR-6.3.4), не обязательно ближайший
- **FR-6.3.2**: Инженер может переключить подтип на «строительство собственного» (внешние и внутренние линейные)
- **FR-6.3.3**: Результаты анализа сохраняются в базе данных для расчёта базового варианта
- **FR-6.3.4**: API списка кандидатов для внешних подтипов: `GET .../pois/:poiId/candidates?subtype=&limit=` — объекты проекта, сортировка по geodesic-расстоянию (см. [architecture.md](../architecture/architecture.md))
- **FR-6.3.5** *(Could Have)*: Для внутренних линейных допускается ручное переопределение `km_per_pad` и/или итогового `distance_km` (без выбора объекта на карте в MVP)

## 7. Расчёт стоимости анализа

- **FR-7.1.1**: На основе анализа окружения система рассчитывает стоимость для точки интереса: для **внешних** — ближайшие Point-объекты; для **внутренних линейных** — `distance_km` по FR-5.3.4; кустовые площадки — FR-5.3.1
- **FR-7.1.2**: Расчёт учитывает все инженерные параметры точки интереса
- **FR-7.2.1**: **Линейные внутренние подтипы** (Автодорога, Нефтепровод, Водопровод, ЛЭП): `distance_km = pads_count × km_per_pad` (FR-5.3.4); стоимость = `distance_km` × ставка (**тыс. ₽/км** из FR-4.1.2); в матрице и отчётах — **млн ₽**. При `not_required` — стоимость 0
- **FR-7.2.2**: **Площадные внешние подтипы** (ГКС, ГТЭС, ПС/ТП, НПЗ): стоимость = фиксированная ставка (**тыс. ₽** за объект). Расстояние не влияет на стоимость.
- **FR-7.2.3**: **Кустовые площадки**: стоимость = количество (шт., округление вверх) × ставка (**тыс. ₽/шт.**).
- **FR-7.2.4**: **Инженерное оборудование** рассчитывается по 5 параметрам:
  - Электроснабжение: "Внешнее" = 0, "Внутреннее" = ставка ГТЭС/ГПЭС
  - Закачка: "Централизованная" = 0, "Локальная" = ставка локальной насосной
  - Утилизация газа: "В пласт" / "Факел" = 0, "Электрогенерация" = ставка ГПЭС/ГТУ (только при "Внутреннее" электроснабжении)
  - Подготовка нефти: МФНС = 0; МКОС / БМУПН / ЦПС(УПН) / УПСВ = соответствующая ставка из FR-4.1.2
  - Транспортировка: все варианты = 0 (т.к. стоимость трубопровода учитывается в подтипе Нефтепровод)
  - Общая стоимость инженерного оборудования = сумма по 5 параметрам (электро, закачка, газ, подготовка нефти, транспортировка)
- **FR-7.2.5**: Общая стоимость = сумма стоимостей всех активных подтипов + стоимость инженерного оборудования

## 8. Матрица решений для точки интереса

### 8.1 Структура матрицы
- **FR-8.1.1**: Матрица отображается в **вертикальном формате** (таблица по умолчанию): строки = параметры, **столбцы = точки интереса** проекта; допускается **карточный вид** для обзора POI
- **FR-8.1.2**: Параметры разделены на визуальные группы: **внутренние подтипы**, **инженерные параметры (бейджи)**, **инженерное оборудование (стоимость)**, **внешние объекты**
- **FR-8.1.3**: Внутренние параметры: Флюид, Объём добычи, Объём закачки (вода), **Сбор продукции скважин**, Автодорога, Нефтепровод, Водопровод, ЛЭП, Кустовые площадки, Электроснабжение, Закачка, Утилизация газа, **Подготовка нефти**, Транспортировка
- **FR-8.1.4**: Внешние параметры: ГКС, ГТЭС, ПС/ТП, НПЗ

### 8.2 Отображение инженерных параметров
- **FR-8.2.1**: Инженерные параметры отображаются как **группы бейджей**: все доступные значения серым, выбранное подсвечивается цветом
- **FR-8.2.2**: Цветовая кодировка: Электроснабжение — синий, Закачка — фиолетовый, Утилизация газа — оранжевый, Подготовка нефти — изумрудный, Сбор скважин — бирюзовый, Транспортировка — серый
- **FR-8.2.3**: Подтипы отображаются с расстоянием и статусом; **внешние** — с именем выбранного/ближайшего объекта; **внутренние линейные** — с формулой км/КП × число КП (FR-5.3.4)

### 8.3 Взаимодействие
- **FR-8.3.1**: Инженер может изменить инженерный параметр в матрице — система автоматически пересчитывает доступность связанных подтипов
- **FR-8.3.2**: Инженер может отфильтровать точки интереса с превышениями
- **FR-8.3.3**: При выборе столбца POI в матрице на карте подсвечиваются линии подключения

## 10. Визуализация на карте

### 10.1 Иконки объектов
- **FR-10.1.1**: Каждый подтип инфраструктуры имеет уникальную цветную иконку на карте (**Lucide** в приложении)
- **FR-10.1.2**: Иконки круглые, с белой рамкой и тенью
- **FR-10.1.3**: При наведении иконка увеличивается на 15%
- **FR-10.1.4**: Точки интереса отображаются крупнее остальных объектов

### 10.2 Пороговые радиусы
- **FR-10.2.1**: Вокруг точки интереса отображаются полупрозрачные круги (opacity 0.15) с пороговыми расстояниями
- **FR-10.2.2**: Каждый радиус имеет цвет, соответствующий типу инфраструктуры
- **FR-10.2.3**: Пользователь может индивидуально включать/выключать отображение каждого радиуса через панель слоёв
- **FR-10.2.4**: Радиусы не перекрывают объекты на карте

### 10.3 Линии подключения
- **FR-10.3.1**: При выборе точки интереса на карте отображаются линии от POI к **якорям внешних** объектов (`anchor_geometry`). Внутренние линейные подтипы линии подключения на карте в MVP **не отображают** (нет якоря)
- **FR-10.3.2**: Цвет линии соответствует статусу: зелёный (в норме), красный (превышение), янтарный (строительство)
- **FR-10.3.3**: При переключении между точками интереса линии обновляются
